minimum fix scope:
src/cli.ts and package.json

repro:

## high: Stale Unix socket leaves the CLI unable to restart the daemon after a crash

id: fnd_sig-feat-cli-command-012418b101-_95c4a24060
category: bug
confidence: high
triage: confirmed-bug
status: fixed
feature: Node source src (feat_cli-command_012418b101)

evidence:
- src/ipc/client.ts:17-20 (request)
- src/ipc/server.ts:7-8 (serve)

Auto-start is gated only on the socket path being absent. On Unix, a daemon crash or SIGKILL typically leaves the socket file on disk, so subsequent requests skip `ensureDaemon()`, attempt to connect to a dead socket, and fail permanently. The stale socket is only cleaned up inside `serve()`, but that code never runs because restart is skipped.

recommendation:
Make `request()` recover from stale sockets by attempting a ping/connect first and falling back to `ensureDaemon()` on connect errors such as `ECONNREFUSED`/`ENOENT`, or always call `ensureDaemon()` when autostart is enabled and let its ping logic decide whether a restart is needed.

test analysis:
The only tests exercise `parseDuration`, `EventQueue`, and `isLoopbackAddress`; there is no daemon lifecycle or crash-recovery coverage.

suggested regression test:
An integration test that starts the daemon, force-kills it without removing the socket file, then verifies a fresh `request('ping')` respawns the daemon and succeeds.

minimum fix scope:
src/ipc/client.ts and, if needed, the daemon startup/cleanup path in src/ipc/server.ts

repro:
Start the daemon, kill it with SIGKILL so `daemon.sock` remains, then run any command that calls `request()`. It fails on `net.createConnection(socketPath())` instead of respawning the daemon.

## medium: `--url` is interpolated into HTML without escaping, allowing script injection in `prompt`/`open`

id: fnd_sig-feat-cli-command-0f8d81242f-_ac54fef201
category: security
confidence: high
triage: confirmed-bug
status: fixed
feature: CLI command glimpse (feat_cli-command_0f8d81242f)

evidence:
- src/cli.ts:24-25

Both `prompt` and `open` build an `<iframe src="...">` wrapper by concatenating the raw `o.url` string directly into an HTML attribute. `assertUrlAllowed` only classifies the URL; it does not sanitize or normalize the original string before that same raw value is rendered. A URL containing a quote can break out of the `src` attribute and inject arbitrary HTML/JS into the wrapper document. In `prompt`, that injected code runs in the bridged page context because `withBridge(...)` is always applied and the default CSP permits inline scripts.

recommendation:
Construct the iframe with an escaped/serialized URL value, e.g. by parsing once with `new URL(...)` and interpolating `url.href`, or by HTML-escaping attribute values before insertion. Add a regression test that passes quotes in `--url` and asserts the generated HTML keeps them inside the `src` value.

test analysis:
No linked tests are present for the CLI entrypoint, and there is no evidence of a regression test covering hostile `--url` input.

suggested regression test:
Add a CLI-level test for `prompt --url` and `open --url` that stubs the downstream call, passes a URL containing quotes, and asserts the generated HTML contains a safely escaped `src` attribute with no injected attributes or tags.

minimum fix scope:
src/cli.ts

repro:
Run `glimpse prompt --url 'http://localhost/" onload="alert(1)'` and the injected handler is rendered as part of the wrapper HTML instead of remaining confined to the iframe URL.

minimum fix scope:
src/daemon/daemon.ts, with any needed registry helper changes in src/daemon/window-registry.ts

repro:
Call `wait` on a nonexistent window ID, or on a closed window after its final event has already been consumed, without `--timeout`. The command never returns.

## medium: Cold-start race can spawn multiple daemons concurrently

id: fnd_sig-feat-cli-command-17d514f2b2-_97e1ce9843
category: concurrency
confidence: high
triage: confirmed-bug
status: fixed
feature: Node package glimpse-cli (feat_cli-command_17d514f2b2)

evidence:
- src/ipc/client.ts:8-13 (ensureDaemon)
- src/ipc/server.ts:8 (serve)
- src/ipc/server.ts:25 (serve)

`ensureDaemon()` attempts to create a lock directory but ignores failure and spawns anyway. Two CLI processes starting from a cold state will both see `ping()` fail and both launch a daemon. Each daemon then unconditionally unlinks and binds the same socket path, so one process can steal the socket path from the other and leave an orphaned daemon behind. The lock path currently does not serialize startup at all.

recommendation:
Make startup single-writer: if lock acquisition fails, wait for the existing starter to finish instead of spawning. Also treat socket bind failure or unexpected socket replacement as a startup error rather than silently racing.

test analysis:
There is no parallel startup test and no assertion that a cold start yields exactly one daemon process.

suggested regression test:
Add an integration test that fires two concurrent requests from a clean state and asserts only one daemon instance is created and both requests succeed against the same socket.

minimum fix scope:
src/ipc/client.ts startup locking and daemon single-instance handling in src/ipc/server.ts

repro:
From a clean runtime directory, launch two CLI commands that autostart the daemon at the same time and inspect the resulting daemon processes/socket behavior.

## medium: Daemon liveness is inferred from socket-file existence, so stale sockets break commands instead of recovering

category: bug
confidence: high
triage: confirmed-bug
status: fixed
feature: Node package glimpse-cli (feat_cli-command_17d514f2b2)

evidence:
- src/ipc/client.ts:17-18 (request)
- src/cli.ts:32
- src/platform/paths.ts:11 (socketPath)

Both autostart and `list` use `existsSync(socketPath())` as the liveness check. If the daemon crashes or is killed and leaves `daemon.sock` behind, `open`/`send`/etc. will skip `ensureDaemon()`, `list` will skip the `running:false` fallback, and every command will fail on connect instead of restarting or reporting a stopped daemon. Because the socket path is stable, one stale file is enough to wedge the CLI until manual cleanup.

recommendation:
Use an actual ping/connect probe for liveness, not file existence. On connection errors such as `ENOENT`/`ECONNREFUSED`, remove the stale socket if appropriate and retry through `ensureDaemon()` before surfacing failure.

test analysis:
Included tests only cover duration parsing, event queue behavior, and loopback detection. There is no daemon crash-recovery or stale-socket integration test.

suggested regression test:
Create a stale socket scenario, then assert both `request('open')` and `glimpse list` recover by starting a new daemon or return a clean `running:false` result.

minimum fix scope:
src/ipc/client.ts and the `list` command path in src/cli.ts

repro:
Start the daemon, kill it uncleanly so `daemon.sock` remains, then run `glimpse list` or `glimpse open ...`. The command fails on socket connection instead of recovering or reporting `running:false`.
